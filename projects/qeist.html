The problem

QA teams receive a pull request with no context about what actually changed. They either re-test everything, or miss the one thing that broke.

What I built

A tool that reads the code changes in a PR and generates the QA checklist for it, so nobody writes test cases by hand. Solo across the whole thing — backend, landing page, a demo repository (shopflow-backend), and the outreach pipeline — with Claude Code as the primary build agent.

The outcome

Interviewed 10 QA engineers working in industry. Every one said they'd use it. Building v2 on what they told me was missing.

“Ten relevant test cases are better than fifty generic ones — if Qeist can do that consistently, teams will pay for it.”Senior QA Engineer, MNC · 5+ yrs · independently interviewed
